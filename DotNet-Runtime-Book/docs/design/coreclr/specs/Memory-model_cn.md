
# .NET 内存模型

## ECMA 335 与 .NET 内存模型
ECMA 335 标准定义了非常弱的内存模型。经过二十年，希望拥有灵活模型的诉求并未带来显著收益，因为硬件实际上更严格。另一方面，针对 ECMA 模型编程需要额外复杂性，以处理难以理解且无法测试的场景。

在多个版本的过程中，.NET 运行时实现逐渐形成了一种内存模型，它是在当前硬件上可高效实现与对开发者相对易于理解之间的实际折衷。本文阐释 .NET 运行时在当前实现中提供并期望的那些不变性，并期望这些不变性会沿用到未来版本。

## 对齐
当由 .NET 运行时管理时，内置原始类型变量会根据其数据类型大小进行*正确对齐*。这适用于堆和栈分配的内存。

1 字节、2 字节、4 字节变量分别存放在 1 字节、2 字节、4 字节边界上。
在 64 位平台上，8 字节变量按 8 字节对齐。
本机大小的整数类型和指针对齐方式与其在给定平台上的大小一致。

当使用 `FieldOffsetAttribute` 显式调整字段偏移时，不保证字段的对齐。

## 原子内存访问
对大小不超过平台指针大小的原始类型和枚举类型的*正确对齐*数据的内存访问始终是原子的。观察到的值始终是完整读写操作的结果。

原始类型：bool, char, int8, uint8, int16, uint16, int32, uint32, int64, uint64, float32, float64, native int, native unsigned int.

非托管指针的值被视为本机整数原始类型。对*正确对齐*的非托管指针值的内存访问是原子的。

托管引用始终按其在给定平台上的大小对齐，访问是原子的。

当变量的位置由运行时管理时，以下方法在所有平台上都执行原子内存访问。
- `System.Threading.Interlocked` 方法
- `System.Threading.Volatile` 方法

**示例：** 在 32 位平台上，`Volatile.Read<double>(ref location)` 是原子的，而对 `location` 的普通读取可能不是。

## 非托管内存访问
由于非托管指针可以指向任何可寻址内存，使用此类指针的操作可能违反运行时提供的保证，并暴露未定义或平台特定的行为。
**示例：** 通过目标地址未按数据访问大小*正确对齐*的指针进行内存访问，可能不是原子的，或会因平台与硬件配置而导致故障。

虽然不常见，未对齐访问仍是现实场景，因此对未对齐内存访问有一些有限支持，例如：
* `unaligned.` IL 前缀
* `Unsafe.ReadUnaligned`, `Unsafe.WriteUnaligned` 和 ` Unsafe.CopyBlockUnaligned` 帮助方法。

这些机制确保对可能未对齐的位置进行无故障访问，但不保证原子性。

截至本文撰写时，尚无针对非一致性内存、设备内存或类似内存的特定支持。通过指针操作或本机互操作把非普通内存传递给运行时会导致未定义行为。

## 内存访问的副作用与优化
.NET 运行时假定内存读写的副作用仅包括在指定内存位置观察和改变值。适用于所有读写——无论是否 volatile。**这与 ECMA 模型不同。**

因此：
* 不允许推测性写入。
* 不能引入读取。
* 未使用的读取可以被消除。（注意：如果读取可能导致故障，则不算“未使用”）
* 可以合并来自同一位置的相邻非 volatile 读取。
* 可以合并对同一位置的相邻非 volatile 写入。

这些规则的现实动机是：
- 我们不能允许推测性写入，因为我们认为改变值是可观察的，因此推测性写入的影响可能无法撤销。
- 读取不能被重新执行，因为它可能取到不同的值，从而引入程序原本不存在的数据竞争。
- 读取变量而不观察其读取副作用等同于未执行读取，因此未使用的读取可以删除。
- 合并对同一位置的相邻普通内存访问是可以的，因为大多数程序不依赖数据竞争的存在；因此与引入数据竞争不同，移除数据竞争是可以的。确实依赖观察数据竞争的程序应使用 `volatile` 访问。

## 线程本地内存访问
优化编译器可能证明某些数据只被单线程访问。在这种情况下，允许进行进一步优化，例如复制或移除内存访问。

## 跨线程访问局部变量
- 不存在类型安全的机制来从一个线程访问另一个线程栈上的位置。
- 通过不安全代码访问位于其他线程栈上的托管引用会导致未定义行为。

## 内存操作的顺序
* **普通内存访问**
普通读写的效果可以被重排序，只要保持单线程一致性。这样的重排序可能来自编译器的代码生成策略，或硬件的弱内存排序。

* **Volatile 读取** 具有“获取语义”——程序顺序中位于其后的读写不得在 volatile 读取之前被推测执行。
  具有获取语义的操作：
     - 带 `volatile.` 前缀且该指令支持此前缀的 IL load 指令
     - `System.Threading.Volatile.Read`
     - `System.Thread.VolatileRead`
     - `System.Threading.Volatile.ReadBarrier`（作用于所有先前读取）
     - 获取锁（`System.Threading.Monitor.Enter` 或进入同步方法，作用于所有先前读取）

* **Volatile 写入** 具有“释放语义”——volatile 写入的效果在程序顺序中之前的所有读写效果可被观察到之前，不得被观察到。
  具有释放语义的操作：
     - 带 `volatile.` 前缀且该指令支持此前缀的 IL store 指令
     - `System.Threading.Volatile.Write`
     - `System.Thread.VolatileWrite`
     - `System.Threading.Volatile.WriteBarrier`（作用于所有后续写入）
     - 释放锁（`System.Threading.Monitor.Exit` 或离开同步方法，作用于所有后续写入）

* **volatile. initblk** 具有“释放语义”——`.volatile initblk` 的效果不会早于其之前的读写效果被观察到。

* **volatile. cpblk** 在读取和写入的内存位置上结合了 volatile 读取与写入的排序语义。
     - `volatile. cpblk` 执行的写入不会早于之前读写的效果被观察到。
     - 程序顺序中位于其后的读写不得在 `volatile. cpblk` 执行的读取之前被推测执行
     - `cpblk` 可以被实现为一系列读写，其粒度和相互顺序未指定。

注意，volatile 语义本身并不意味着操作是原子的，也不影响操作提交到一致性内存的时间。它只规定这些效果最终被观察到时的顺序。

`volatile.` 与 `unaligned.` IL 前缀在两者均被允许的情况下可以组合使用。

优化编译器可能证明某些数据仅由单线程访问。在这种情况下，访问这些数据时允许省略 volatile 语义。

* **全栅栏操作**
  全栅栏操作具有“全栅栏语义”——读写效果必须不早于也不晚于全栅栏操作被观察到，且遵循其相对程序顺序。
  具有全栅栏语义的操作：
     - `System.Thread.MemoryBarrier`
     - `System.Threading.Interlocked` 方法

> ### 内存的唯一副作用
>
> > “.NET 运行时假定内存读写的副作用仅包括在指定内存位置观察和改变值。”
>
> 即：编译器认为，读写一个变量，仅仅就是读写这个变量本身，不会像魔法一样引发其他隐藏的后果（除非你显式加了锁或调用了方法）。因此，编译器大前提是：**只要最终结果对得上，中间过程我可以随便改。**
>
> 但为了防止多线程下的灾难，必须制定以下 5 条铁律：
>
> ### 铁律 1：不允许推测性写入（No speculative writes）
>
> - 场景：假设你有这样的 C# 代码：
>
>   ```c#
>   if (condition) {
>       sharedVar = 1;
>   } else {
>       sharedVar = 2;
>   }
>   ```
>
> - **如果不遵守这条规则（坏的编译器优化）：** 编译器为了速度，可能会先“盲猜”一个结果写进去，发现错了再改： `sharedVar = 1; // 先不管三七二十一写个1` `if (!condition) sharedVar = 2; // 如果条件不成立，再改回2`
>
> - **为什么被禁止？** 在单线程下这没毛病。但在多线程下，如果 `condition` 明明是 `false`，另一个线程却可能在极短的瞬间读到了那个错误的 `1`！**写内存是泼出去的水，其他线程立刻就能看到，绝不能瞎写。**
>
> ### 铁律 2：不能引入读取（No introduced reads）
>
> - 场景：你的 C# 代码里明明只读了一次变量：
>
>   ```c#
>   int temp = sharedVar; // 读一次
>   if (temp > 0) {
>       DoSomething(temp); // 用刚才读到的值
>   }
>   ```
>
> - **如果不遵守这条规则（坏的编译器优化）：** 编译器觉得没必要分配寄存器存 `temp`，直接每次都去内存里读： `if (sharedVar > 0) { DoSomething(sharedVar); } // 读了两次！`
>
> - **为什么被禁止？** 这会导致致命的 **TOCTOU（Time-of-check to time-of-use）漏洞**。如果在 `if` 检查通过后，瞬间另一个线程把 `sharedVar` 改成了 `-100`，那么 `DoSomething(-100)` 就会被执行，程序逻辑彻底崩溃！所以，代码里写了读几次，底层就只能读几次，绝不能多读。
>
> ### 铁律 3：未使用的读取可以被消除
>
> - **场景：** 你写了无用的代码：`int a = x;`，但后面根本没用到 `a`。
> - **规则含义：** 编译器可以直接把这行代码删了（死代码消除）。
> - **特殊例外（可能导致故障的读取）：** 如果你写的是 `int a = obj.Field;`，即使 `a` 没被用到，只要 `obj` 有可能是 `null`，编译器就**不能删**这行代码。因为它必须保留触发 `NullReferenceException` 崩溃的权利，这是 C# 语义的一部分。
>
> ### 铁律 4 & 5：可以合并相邻的非 volatile 读/写
>
> - **合并读（合并读取）：** 你写了：`int a = sharedVar; int b = sharedVar;` 编译器优化为：`int a = sharedVar; int b = a;`（只去内存读一次，第二次直接用 CPU 寄存器里的缓存）。
> - **合并写（合并写入）：** 你写了：`sharedVar = 1; sharedVar = 2;` 编译器优化为：直接跳过 1，只执行 `sharedVar = 2;`。
> - **为什么这是合法的？** 因为对于**普通变量（非 volatile）**，编译器假设当前没有其他线程在跟你抢这个变量。合并它们不仅提高了性能，还**减少了多线程并发时的冲突窗口**。
> - **关键字 `volatile` 的意义：** 文档最后一句点明了主旨——如果你**真的**写了一个自旋锁（比如 `while(!flag) {}`），你就是想看另一个线程什么时候把 `flag` 变成 `true`，那你**必须加 `volatile`**。如果不加，编译器就会根据规则 4，把代码优化成死循环（只读一次 `flag`，发现是 false，就永远循环下去了）。
>
> 
>
> **CPU 和 JIT 编译器为了提升性能，会疯狂地把你写的代码顺序打乱执行（指令重排）。在单线程里这没问题，但在多线程里，这会导致致命的并发 Bug。**
>
> 为了镇住 CPU 和编译器，.NET 提供了以下几种“武器”：
>
> ### 1. 普通内存访问
>
> 保持单线程一致性的前提下可以被重排序。
>
> ### 2. Volatile 读取 = 获取语义 (Acquire Semantics) = “向下单向阀门”
>
> 位于其后的读写不得在 volatile 读取之前被推测执行。
>
> `Volatile.Read` 就像一道**只允许代码往下沉，不允许代码往上浮的单向阀门**。
>
> - **核心铁律：** 写在它**后面**的代码，**绝对不允许**被 CPU 重排到它的**前面**去执行。
> - **典型场景：** 进锁（`Monitor.Enter`）。
> - **想一想为什么：** 你加锁是为了保护后面的代码。如果 CPU 把后面的代码优化到了加锁的前面去执行，锁不就白加了吗？所以进锁必须是“获取语义”。
>
> ### 3. Volatile 写入 = 释放语义 (Release Semantics) = “向上单向阀门”
>
> `Volatile.Write` 就像一道只允许代码往上浮，不允许代码往下沉的单向阀门。
>
> - **核心铁律：** 写在它**前面**的代码，**必须全部执行完**并且刷新到主内存，才能执行这句写入。前面的代码**绝对不允许**掉到它的**后面**。
> - **典型场景：** 解锁（`Monitor.Exit`）、或者给一个 `isReady = true` 的标志位赋值。
> - **想一想为什么：** 假设你组装了一个复杂的对象，最后一步把 `isReady` 设为 `true`。如果 CPU 把 `isReady = true` 重排到了组装对象的前面，别的线程看到 `true` 就去拿对象，结果拿到一个半成品，程序直接崩溃。所以这种写入必须是“释放语义”。
>
> ### 4. 全栅栏操作 (Full Fence) = “承重墙”
>
> 顾名思义，这是一堵承重墙。
>
> - 上面的代码绝对下不来，下面的代码绝对上不去。一切重排到此为止。
> - **典型场景：** 所有的 `Interlocked` 操作（如 `Interlocked.Increment`）、`Thread.MemoryBarrier()`。
> - **代价：** 极其沉重，它会强制清空 CPU 的流水线和缓存队列。非必要不使用。

## C# `volatile` 特性
引入 volatile 内存访问的一种常见方式是使用 C# 的 `volatile` 语言特性。将字段声明为 `volatile` 不会改变 .NET 运行时如何处理该字段。该修饰只是给 C# 编译器（以及其他 .Net 语言的编译器）的提示，使其对该字段的读写发出带 `volatile.` 前缀的读写。

## 进程范围屏障
进程范围屏障具有全栅栏语义，并额外保证：程序中的每个线程都会在与进程范围屏障同步的任意点上有效执行一次全栅栏，使得在两个屏障之前的写入效果可被之后的内存操作观察到。

具体实现可能依平台而异。例如，中断当前进程亲和性掩码中每个核心的执行就是一种可行的实现。

## 同步方法
使用 ```MethodImpl(MethodImplOptions.Synchronized)``` 特性修饰的方法，其内存访问语义等同于在方法入口处获取锁并在离开时释放锁。

## 数据依赖读取有序
内存排序遵循数据依赖。当从由引用派生出的地址进行间接读取时，保证读取数据不会先于获取该引用发生。这一保证同时适用于托管引用和非托管指针。

**示例：** 读取字段时，不会使用在获得实例引用之前从该字段位置取得的缓存值。

 ```cs
var x = nonlocal.a.b;
var y = nonlocal.a;
var z = y.b;

// 不可能出现如下执行顺序：

var x = nonlocal.a.b;
var y = nonlocal.a;
var z = x;
 ```

## 对象赋值
将对象赋值到可能被其他线程访问的位置，相对于该实例的字段/元素和元数据的访问而言是一次 release。优化编译器必须保持对象赋值与数据依赖内存访问的顺序。

其动机是确保将对象引用存入共享内存时充当一个“提交点”，以覆盖通过该实例引用可达的所有修改。还保证当其他线程（包括后台 GC 线程）能够访问该实例时，刚分配的实例是有效的（例如方法表和必要的标志已设置）。
读取线程在访问实例内容之前不需要执行获取式读取，因为运行时保证数据依赖读取的排序。

引用赋值的排序副作用不应被用于通用排序目的，因为：
- 独立的非 volatile 引用赋值可能被编译器重排序。
- 如果优化编译器能证明实例未与其他线程共享，则可以省略 release 语义。

关于对象赋值所提供的保证曾有诸多歧义。今后运行时只提供本文描述的保证。

_据信编译器优化不会违反 [data-dependent reads](#data-dependent-reads-are-ordered) 和 [object assignments](#object-assignment) 部分的排序保证，但仍需进一步调查以确保符合或修复可能的违规。这由以下问题跟踪：_ https://github.com/dotnet/runtime/issues/79764

## 实例构造函数
.NET 运行时不对实例构造函数指定任何排序效应。

## 静态构造函数
静态构造函数执行的所有副作用不晚于访问该类型任何成员的效果被观察到。对该类型其他成员方法的调用将观察到静态构造函数执行的完整结果。

## 硬件考虑
当前支持的 .NET 运行时及系统库实现对硬件内存模型有若干假设。这些条件在所有支持的平台上都成立，并对运行时用户透明地传递。未来支持的平台也很可能支持这些条件，因为庞大的既有软件使得打破常见假设代价高昂。

* 自然对齐且大小不超过平台指针大小的读写是原子的。
这也适用于被不同大小的重叠对齐读写所作用的位置。
**示例：** 对 4 字节对齐的 int32 变量读取将得到某次写入之前或之后的值，而不会是前后字节的混合。

* 内存是缓存一致的，对单一位置的写入会被所有核心以相同顺序观察到（多副本原子性）。
**示例：** 当同一位置按升序更新（例如 1,2,3,4,...）时，没有观察者会看到降序序列。

* 只要不违反单线程一致性，线程可能在其他核心看到之前先看到自己的写入（存储缓冲转发）。

* 运行时管理的内存是普通内存（非设备寄存器文件等），内存操作的唯一副作用是存储和读取值。

* 可以实现释放一致性内存模型。
要么平台默认就是释放一致性或更强（例如 x64 为 TSO，更强），要么平台提供通过栅栏操作实现释放一致性的手段。

* 可以保证数据依赖读取的顺序。
要么平台默认遵循数据依赖（当前所有支持的平台均如此），要么平台提供通过栅栏操作对数据依赖读取排序的手段。

## 示例与常见模式
以下示例在所有支持的 .NET 运行时实现上都能正确工作，与目标操作系统或体系结构无关。

*   构造实例并与其他线程共享是安全的，不需要显式栅栏。

```cs

static MyClass obj;

// 线程 #1
void ThreadFunc1()
{
    while (true)
    {
        obj = new MyClass();
    }
}

// 线程 #2
void ThreadFunc2()
{
    while (true)
    {
        obj = null;
    }
}

// 线程 #3
void ThreadFunc3()
{
    MyClass localObj = obj;
    if (localObj != null)
    {
        // 访问本地对象的成员是安全的，因为
        // - 不能引入读取，因此 localObj 不会被重新读取并变为 null
        // - 对 obj 的发布赋值不会早于 MyClass 构造函数中的写操作被观察到
        // - 通过实例进行的间接访问是数据依赖读取，因此我们会看到构造函数写入的结果
        System.Console.WriteLine(localObj.ToString());
    }
}

```

* Singleton（使用锁）

```cs
public class Singleton
{
    private static readonly object _lock = new object();
    private static Singleton _inst;

    private Singleton() { }

    public static Singleton GetInstance()
    {
        if (_inst == null)
        {
            lock (_lock)
            {
                // 获取锁具有获取语义，读取 _inst 会发生在获取锁之后
                // 释放锁具有释放语义，若其他线程赋值了 _inst，该写入最晚会在释放锁时被观察到
                // 因此如果其他线程已初始化 _inst，当前线程在此处保证能看到。

                if (_inst == null)
                {
                    _inst = new Singleton();
                }
            }
        }

        return _inst;
    }
}

```

* Singleton（使用 interlocked 操作）

```cs
public class Singleton
{
    private static Singleton _inst;

    private Singleton() { }

    public static Singleton GetInstance()
    {
        Singleton localInst = _inst;
        if (localInst == null)
        {
            // 不同于使用锁的示例，我们可能构造多个实例
            // 只有一个会“胜出”并成为唯一的单例对象
            Interlocked.CompareExchange(ref _inst, new Singleton(), null);

            // 由于 Interlocked.CompareExchange 是全栅栏，
            // 我们不可能读到 null 或其他不是单例的伪实例
            localInst = _inst;
        }

        return localInst;
    }
}
```

* 通过检查标志与另一个线程通信

```cs
internal class Program
{
    static bool flag;

    static void Main(string[] args)
    {
        Task.Run(() => flag = true);

        // 重复读取最终会看到 'flag' 的值已改变，
        // 但读取必须是 Volatile，以确保所有读取不会被合并
        // 在进入 while 循环之前被合并成一次读取。
        while (!Volatile.Read(ref flag))
        {
        }

        System.Console.WriteLine("done");
    }
}
```

> ### 2. Singleton（使用锁 - Double Check Locking）
>
> ```
> if (_inst == null) {
>     lock (_lock) {
>         if (_inst == null) {
>             _inst = new Singleton();
>         }
>     }
> }
> ```
>
> 这是经典的“双重检查锁定”。结合上一篇文档的知识，注释的意思是：
>
> - **进锁（获取语义）：** 保证只要进去了，你读到的 `_inst` 绝对是主内存里最新鲜的，而不是你当前线程缓存里的过期数据。
> - **出锁（释放语义）：** 保证如果线程 A `new` 了一个对象并赋值给 `_inst`，等它退出 lock 时，这个新对象会立刻被“推”到主内存。别的线程绝对能看到。
>
> ### 3. Singleton（使用 Interlocked 操作 - 无锁高并发）
>
> ```
> Interlocked.CompareExchange(ref _inst, new Singleton(), null);
> ```
>
> 在极高并发下，`lock` 太重了，可以用这种无锁（Lock-free）写法。
>
> - **它怎么工作？** 100个线程同时跑过来，发现 `_inst == null`，于是这 100 个线程**同时**去堆上 `new Singleton()`（产生了 100 个对象）。
> - **决胜时刻：** 到了 `Interlocked.CompareExchange` 这堵**“全栅栏（承重墙）”**。它是硬件级别的原子操作。100个线程同时想把自己的对象塞给 `_inst`，但只有最快的那个能成功。剩下的 99 个线程会发现 `_inst` 已经不是 `null` 了，操作失败。它们创建的 99 个废弃对象直接被丢弃，等 GC 来收垃圾。
> - **安全性：** 因为有全栅栏，不会出现“读取到一半”的幽灵状态。
>
> ### 4. 通过检查标志与另一个线程通信 (Volatile 旋转)
>
> ```
> while (!Volatile.Read(ref flag)) { }
> ```
>
> 这就是你在等快递，一直刷手机（自旋锁）。
>
> - 为什么不用普通的 `while(!flag)`？
>
> - 因为上一篇文档说了：**“可以合并来自同一位置的相邻非 volatile 读取”**。
>
> - 如果你写 `while(!flag)`，JIT 编译器会觉得：“你在这个循环里又没改 flag，我干嘛每次都去内存读？” 于是 JIT 会把它优化成：
>
>   ```
>   bool temp = flag;
>   while (!temp) { } // 如果一开始是 false，就成了死循环！
>   ```
>
> - 加了 `Volatile.Read`（向下单向阀门），就是用枪指着编译器的脑袋说：“**不许合并！不许优化！老老实实每次都去给我查主内存！**”
